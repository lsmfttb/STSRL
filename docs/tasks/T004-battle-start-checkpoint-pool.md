# T004: Battle-Start Checkpoint Pool

## Objective

Create restorable, auditable natural battle-start checkpoints generated by
explicit controllers. This is the source foundation for stratified training
and fixed battle evaluation.

## Current Main Baseline

`main` has the authoritative `execute_controlled_run` path, native
process-local checkpoint restore, portable battle-start manifests, complete
controller provenance, and the named stochastic non-combat driver.

## Dependencies

- T002 is complete: controlled execution and controller provenance.
- T003 is complete: versioned artifacts and migration rules.
- T010 is complete: named stochastic natural-run driver and natural-coverage
  reporting.

## Scope

- Add authoritative `sts_lightspeed` checkpoint capture and restore support
  through repository patch files and the Python adapter.
- Verify deterministic replay from captured battle starts.
- Collect one checkpoint for every naturally reached battle in a controlled
  run.
- Persist structural metadata including ascension, act, room type, encounter
  identity, floor, source run, source battle index, and controller provenance.
- Support seeded structural sampling while retaining natural-source identity
  and sampling component.
- Audit coverage separately from wins and progression.
- Add explicit schema migration for any supported legacy pool fixture.

Task-owned implementation boundaries are the checkpoint-related
`sts_lightspeed` patch files, the Python simulator adapter, a focused pool and
verification module below `src/sts_combat_rl/sim/`, task-owned tests and
fixtures, and a focused command workflow if one is needed. CLI code remains
limited to parsing and routing.

## Out Of Scope

- Fixed-controller evaluation, search labels, constructed transforms, public
  run history, or model training.
- Hand-written filtering for "reasonable" decks or relics.
- Claiming that repeated samples increase unique-state coverage.
- A battle-policy improvement, constructed battle states, or replacement of a
  visible Boss.

## Design Constraints

- Capture a checkpoint immediately at a naturally reached battle start, before
  the battle controller selects an action. The simulator alone owns capture,
  restore, legal actions, and transitions.
- An opaque checkpoint may contain hidden simulator state. It is a restore
  mechanism, not normal-public-policy input; normal-information consumers
  receive only sanitized public context and explicitly declared information
  regime.
- Every record retains its source run seed, source battle index, source
  controller provenance, natural distribution tag, and sampling component.
  Missing public context or unavailable native fields remain explicit.
- Readers migrate supported pool schemas sequentially before business logic;
  writers emit the current schema only.
- Structural sampling changes optimization weight only. It never changes the
  stored source checkpoint or claims additional unique coverage.

## Acceptance Criteria

- Every fresh checkpoint restores deterministically in a fresh WSL process.
- Every checkpoint records complete generating-controller provenance.
- Natural and structurally sampled records remain distinguishable.
- Coverage reports separate battle starts, unique sources, room types, acts,
  ascensions, and encounters.
- Unsupported or incomplete legacy records fail explicitly.

## Deliverables

- Versioned checkpoint-pool schema, writer, reader, and migration tests.
- Native checkpoint capture/restore adapter surface and source patch files.
- Fresh-process determinism verifier and a focused command workflow.
- Natural pool collector, seeded structural sampler, and coverage report.

## Required Verification

Run all standard local checks plus documented WSL commands for checkpoint
capture, fresh-process restore, small A20 natural-pool collection, and coverage
audit. The verification must name the battle controller, non-combat driver,
ascension, seed range, checkpoint count, unique-source count, and every
reported restore mismatch.
